# MVP Technical Design Document: GNOME Shell Calendar Colors Extension

## 1. Overview

This document describes the minimum viable product for a small GNOME Shell extension that adds visual color distinctions to the calendar popup in the top bar. The goal is to keep the extension small, focused, and compatible with an existing shell styling extension such as Open Bar.

The extension does not redesign the GNOME calendar UI. Instead, it enhances the existing month grid by adding visual markers for three simple cases:

* weekends
* holidays
* days with events

This project is intended to be lightweight, easy to maintain, and suitable for publishing as an open source GitHub repository and later submitting to extensions.gnome.org.

## 2. Goals

The MVP should achieve the following:

* add color-based visual differentiation to calendar day cells
* highlight weekends using a dedicated style
* highlight holidays using a dedicated style
* highlight dates that already have events using a dedicated style or marker
* avoid interfering with the rest of the GNOME Shell theme
* remain compatible with existing visual shell customizations such as Open Bar
* keep the codebase small and easy to review

## 3. Non-Goals

The MVP will not include the following:

* full calendar replacement
* direct integration with Google Calendar APIs or external services
* country-specific public holiday API fetching
* advanced recurrence handling
* drag-and-drop event editing
* complex preferences UI
* animation-heavy visual effects
* deep integration with third-party shell extensions

These items can be considered later after the first release is stable.

## 4. Design Principles

The extension should follow these principles:

### 4.1 Small Surface Area

Only modify the calendar popup day cells. Do not patch unrelated shell components such as the top bar, quick settings, notifications, app grid, or window controls.

### 4.2 Safe Coexistence

Let Open Bar or other theme/styling extensions control the overall look of the shell. This extension should only add semantic classes and small CSS rules for calendar dates.

### 4.3 Minimal Runtime Complexity

Avoid background daemons, unnecessary polling, or network requests. The MVP should work entirely from shell state plus a small locally defined holiday list.

### 4.4 Clear Separation of Responsibilities

* JavaScript handles date classification and class assignment
* CSS handles visual appearance
* optional GSettings handles user-configurable holiday values in later iterations

## 5. User Experience

When the user opens the GNOME top bar calendar popup, the month grid should visually distinguish dates as follows:

* weekends: shown in a different text or background color
* holidays: shown in another distinct color
* days with events: shown using a dot, border, accent, or other clear indicator

If more than one condition applies to the same date, the extension should apply a predictable priority rule.

Recommended priority for MVP:

1. holiday
2. has-event
3. weekend

This means a holiday should visually override a weekend if both conditions are true.

## 6. Functional Scope

## 6.1 Weekend Detection

A date is considered a weekend if its weekday is Saturday or Sunday.

Expected logic:

* Sunday = 0
* Saturday = 6

## 6.2 Holiday Detection

For MVP, holidays are read from a static list of ISO date strings stored locally, for example:

* `2026-01-01`
* `2026-02-11`
* `2026-12-25`

The extension checks whether the displayed calendar day matches one of these strings.

## 6.3 Event Detection

The extension should detect whether a given day already has an event marker or event-related shell state. The exact implementation may depend on the internal GNOME Shell calendar actor structure for the target shell version.

For the MVP, this detection can rely on one of the following:

* existing day actor style classes
* child actors or indicators attached by GNOME Shell
* internal calendar event state already exposed in the popup

The implementation should be kept version-specific and limited to the tested GNOME Shell version(s).

## 7. Technical Architecture

## 7.1 Project Type

This is a GNOME Shell extension using JavaScript and CSS.

For newer GNOME Shell versions, the extension should use the current extension structure expected by GNOME Shell and GJS.

## 7.2 Repository Structure

Recommended GitHub repository layout:

```text
gnome-shell-extension-calendar-colors/
├── metadata.json
├── extension.js
├── stylesheet.css
├── prefs.js
├── schemas/
│   └── org.gnome.shell.extensions.calendarcolors.gschema.xml
├── README.md
├── LICENSE
└── .gitignore
```

For the true MVP, `prefs.js` and `schemas/` can be added after the first working prototype, but it is good to reserve the structure early.

## 7.3 Local Development Layout

The Git repository should live in a normal development directory, for example:

```bash
~/Projects/gnome-shell-extension-calendar-colors
```

GNOME Shell loads extensions from:

```bash
~/.local/share/gnome-shell/extensions/<uuid>/
```

To connect the repository with GNOME Shell during development, use a symbolic link:

```bash
mkdir -p ~/.local/share/gnome-shell/extensions
ln -s ~/Projects/gnome-shell-extension-calendar-colors \
~/.local/share/gnome-shell/extensions/calendar-colors@yourname.github.io
```

This allows a single source of truth for both Git and local testing.

## 8. Core Files

## 8.1 metadata.json

Purpose:

* identifies the extension
* declares supported shell versions
* points to the GitHub repository

Example fields:

```json
{
  "uuid": "calendar-colors@yourname.github.io",
  "name": "Calendar Colors",
  "description": "Adds weekend, holiday, and event colors to the GNOME Shell calendar popup.",
  "shell-version": ["46", "47", "48"],
  "url": "https://github.com/yourname/gnome-shell-extension-calendar-colors",
  "version": 1
}
```

Notes:

* the folder name in the extensions directory must exactly match `uuid`
* only declare shell versions that were actually tested

## 8.2 extension.js

Purpose:

* locate the calendar popup UI
* inspect visible day cells
* determine whether each day is a weekend, holiday, or event day
* add or remove style classes accordingly
* clean up all hooks on disable

Responsibilities:

* attach on `enable()`
* remove all custom effects on `disable()`
* avoid modifying shell UI before enable
* avoid leaving stale classes or signals behind

## 8.3 stylesheet.css

Purpose:

* define visual styling for semantic classes

Example class concepts:

* `.calendar-day-weekend`
* `.calendar-day-holiday`
* `.calendar-day-has-event`

The CSS should remain small and theme-friendly. Avoid hardcoding overly aggressive styles that break dark themes or extensions such as Open Bar.

## 8.4 prefs.js

Purpose:

* later provide a minimal preferences screen
* allow user-defined holiday lists
* optionally allow color customization

This is optional for the first code milestone if the initial prototype uses a hardcoded holiday list.

## 8.5 GSettings Schema

Purpose:

* persist configuration such as:

  * holiday date list
  * weekend color enabled/disabled
  * holiday color enabled/disabled
  * event indicator mode

For the MVP, the first useful schema key is likely:

* `holiday-dates` as an array of strings

## 9. Runtime Flow

## 9.1 Enable Phase

When the extension is enabled:

1. locate the calendar popup actor tree
2. identify the month grid day actors currently visible
3. iterate through the visible day cells
4. compute the date classification for each day
5. apply custom style classes
6. connect any required signals so that the styles refresh when the month view changes

## 9.2 Refresh Phase

The extension should refresh styling when:

* the calendar popup is opened
* the visible month changes
* event indicators change, if detectable through shell signals
* the extension is re-enabled

The MVP should avoid frequent polling. Signal-based refresh is preferred.

## 9.3 Disable Phase

When the extension is disabled:

* disconnect all signals
* remove all custom classes added to day actors
* release references to shell objects
* restore the original shell state as cleanly as possible

This is especially important for GNOME extensions review quality and general stability.

## 10. Date Classification Rules

Each visible day cell should be assigned zero or more semantic states.

### 10.1 Weekend Rule

If `date.getDay()` is `0` or `6`, mark the day as weekend.

### 10.2 Holiday Rule

If the formatted local date string matches one of the entries in the holiday list, mark the day as holiday.

Recommended date format:

```text
YYYY-MM-DD
```

### 10.3 Event Rule

If the day already carries an event indicator from the shell, mark it as has-event.

### 10.4 Priority Rule

If multiple states apply, use this priority:

* holiday overrides weekend
* event may coexist with holiday or weekend if the style is additive
* today should remain clearly visible even when another state is present

Recommended behavior:

* use background color for holiday
* use text color or subtle accent for weekend
* use dot/border/underline for event
* do not remove the built-in “today” distinction

## 11. Styling Strategy

The extension should prefer additive styling rather than replacement styling.

Recommended visual strategy:

* weekend: lighter accent text color
* holiday: stronger background or border color
* has-event: dot, underline, or thin outline
* today: preserve GNOME’s original today marker whenever possible

The design should remain readable in dark mode and under third-party shell themes.

Example conceptual CSS:

```css
.calendar-day-weekend {
  color: rgba(255, 120, 120, 1);
}

.calendar-day-holiday {
  background-color: rgba(255, 160, 60, 0.22);
  border-radius: 9999px;
}

.calendar-day-has-event {
  box-shadow: inset 0 -2px 0 0 rgba(100, 160, 255, 1);
}
```

Exact selectors will depend on the final actor mapping used in the implementation.

## 12. Compatibility Considerations

## 12.1 GNOME Shell Version Changes

GNOME Shell internal actor structures can change between versions. The extension should target a narrow, tested range first.

Recommended initial support:

* one tested GNOME version, or
* a very small version window such as GNOME 46 to 48, only if verified

## 12.2 Open Bar Compatibility

This extension should not try to manage:

* top bar gradients
* popup panel shapes
* menu transparency
* shell-wide palette settings

It should only add semantic styles to date cells. That reduces the chance of conflicts with Open Bar.

## 12.3 Wayland and X11

The extension should behave identically for both, because it operates at GNOME Shell UI level rather than windowing system level.

## 13. Implementation Milestones

## Milestone 1: Static Weekend Styling

* detect visible day actors
* mark Saturdays and Sundays
* apply a distinct style

## Milestone 2: Static Holiday Styling

* add a hardcoded holiday list
* highlight matching dates
* verify priority over weekend styles

## Milestone 3: Existing Event Detection

* inspect GNOME Shell event marker behavior
* add a visual indicator for event days
* ensure it does not interfere with today/highlight state

## Milestone 4: Basic Preferences

* move holiday list into GSettings
* add a small prefs UI for editing holiday dates

## Milestone 5: Packaging and Publishing

* create zip package
* write README
* prepare screenshots
* submit to extensions.gnome.org

## 14. Development Workflow

## 14.1 Create Repository

Create a new GitHub repository such as:

```text
gnome-shell-extension-calendar-colors
```

## 14.2 Start Local Development

Develop inside the repository and symlink it into the GNOME extensions path.

## 14.3 Test Iteratively

Typical loop:

* edit source files
* reload or restart GNOME Shell session as needed
* enable extension
* inspect calendar popup
* verify classes and rendering

Useful commands:

```bash
gnome-extensions enable calendar-colors@yourname.github.io
gnome-extensions disable calendar-colors@yourname.github.io
gnome-extensions info calendar-colors@yourname.github.io
```

## 14.4 Package for Release

Before release, zip the extension contents so that the archive root contains `metadata.json` directly.

Example:

```bash
cd ~/Projects/gnome-shell-extension-calendar-colors
zip -r calendar-colors@yourname.github.io.zip \
metadata.json extension.js stylesheet.css prefs.js schemas README.md LICENSE
```

Do not wrap everything inside an extra top-level folder in the archive.

## 15. Testing Checklist

Before first public release, verify the following:

* extension enables successfully
* extension disables cleanly
* no shell crash or obvious log spam
* weekends are styled correctly
* holidays are styled correctly
* event days are styled correctly
* today remains readable
* month navigation refreshes styles correctly
* extension still works with Open Bar enabled
* dark mode remains readable
* zip package structure is valid

## 16. Risks

## 16.1 Internal Shell API Fragility

GNOME Shell calendar internals are not guaranteed to remain identical across versions. A selector or actor path that works today may change later.

Mitigation:

* keep support versions narrow
* isolate calendar traversal logic
* test after every GNOME upgrade

## 16.2 Theme Conflicts

Third-party shell themes may already style calendar cells aggressively.

Mitigation:

* keep CSS additive and modest
* avoid overriding core shell styles globally

## 16.3 Event Detection Variability

Event-related actor structure may differ depending on shell version.

Mitigation:

* build event detection last
* keep fallback behavior simple
* allow release 1.0 to ship with weekend and holiday support first if needed

## 17. Future Enhancements

After the MVP is stable, possible next steps include:

* editable holiday list in preferences
* country-specific holiday presets
* configurable colors
* custom rule priority
* optional dot vs border event marker styles
* import from local ICS sources
* month legend in the popup
* better localization support

## 18. Conclusion

The best MVP for this project is a small GNOME Shell extension that adds semantic styling to the existing calendar popup rather than replacing the calendar UI. This keeps the project simple, compatible with Open Bar, and realistic for open source maintenance and extension store submission.

The first release should focus on three small outcomes:

* color weekends
* color holidays from a local list
* mark days with events

That scope is small enough to implement cleanly, test properly, publish on GitHub, and evolve later without turning the project into a fragile shell fork.
